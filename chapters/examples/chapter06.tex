\chapter{Diskussion der Ergebnisse}
\label{ch:chapter06}

In diesem Kapitel werden die in Kapitel~\ref{ch:chapter05} vorgestellten Ergebnisse eingeordnet und interpretiert. 
Zunächst werden die zentralen Befunde zu den sprachspezifischen Overheads und absoluten Performances diskutiert. 
Anschließend folgt ein Vergleich mit den verwandten Arbeiten aus Kapitel~\ref{ch:chapter03}. Danach werden die Forschungsfragen 
beantwortet und konkrete Praxisempfehlungen abgeleitet.

\section{Interpretation der zentralen Ergebnisse}

\subsection{Relativer Overhead – ein differenziertes Bild}

Die Rangliste des prozentualen Overheads (Tabelle~\ref{res:ranking-overhead-cold-bootstrap}) zeigt, dass Python über alle 
Metriken und Starttypen hinweg den geringsten Overhead hat (12 erste Plätze von 25). Java liegt mit 8 ersten Plätzen in der Mitte, 
Node.js erreicht nur 5 erste Plätze. Das ist überraschend, da Node.js bei Warmstarts bei der Duration (z.~B. HTTP-GET) einen niedrigeren 
Overhead als Java und sogar Python haben kann (siehe Tabelle~\ref{res:ranking-largest-differences-overhead}). Die Ursache für die 
schlechte Gesamtplatzierung von Node.js sind die extrem hohen Overheads bei der Init Duration von Kaltstarts (bis zu 805\,\% 
bei der POST-Anfrage).

\emph{Warum ist der Init‑Duration‑Overhead bei Node.js so hoch?} 
Eine mögliche Erklärung ist der asynchrone, ereignisgesteuerte Charakter von Node.js. Der ADOT-Layer muss beim Kaltstart zusätzliche 
Hooks in die Event-Loop einbauen, um asynchrone Ressourcen zu verfolgen und den verteilten Kontext zu sichern~\cite{OTelNodeAsyncHooks}. 
OpenTelemetry benutzt dafür die \texttt{AsyncLocalStorage}-API oder, in älteren Versionen, die \texttt{async\_hooks}-API~\cite{OTelNodeAsyncHooks}. 
Die Initialisierung und Registrierung dieser Hooks können ressourcenintensiv sein~\cite{RaphaelMankeOTelESM}. 
Das wird durch Berichte aus der Praxis bestätigt, die zeigen, dass die ADOT-Node.js-Layer zu großen 
Kaltstartverzögerungen führen können~\cite{OTelLambdaColdStartIssue, AWSADOTLayerIssue}.

\subsection{Absolute Performance mit OpenTelemetry}

Die Rangliste der absoluten Messwerte mit OTel (Tabelle~\ref{res:ranking-with-otel}) zeigt ein klares Bild. 
Python ist die schnellste und speichereffizienteste Sprache (24 erste Plätze von 25). Node.js folgt auf dem zweiten Platz 
(24 zweite Plätze), Java ist am langsamsten (23 dritte Plätze). Das ist konsistent mit den Ergebnissen von Rodriguez et 
al.~\cite{Rodriguez2024Evaluation}, die ebenfalls Python als die Sprache mit der besten Performance auf AWS Lambda identifiziert haben. 
Interessant ist der einzige Fall, in dem Python nicht den ersten Platz belegt. Bei Warmstarts der Duration erreicht Java einmal 
den ersten Platz (Benchmark 1 - Lambdaverkettung). 

Eine mögliche Erklärung für die insgesamt beste Performance von Python wäre, dass die Standardbibliothek für HTTP-Anfragen urllib3 sehr viele in C 
implementierte Module auf TLS- / SSL-Ebene verwendet~\cite{Urllib3}. Die AWS-SDKs für Python verwenden stets die urllib3-Bibliothek~\cite{Boto3}. 
Die schlechteste Duration von Python bei der Lambdaverkettung könnte auf die synchrone Natur von Python 
zurückzuführen sein. Während Node.js und Java die Lambdaverkettung asynchron implenetieren können, wird 
bei Python ein Thread-basierter HTTP-Client-Stack verwendet und das ist keine asynchrone Verarbeitung~\cite{Boto3}. 
\subsection{Negative Overheads}

Bei Kaltstarts der HTTP-Benchmarks wurden für Node.js und Python negative prozentuale Overheads gemessen 
(Tabelle~\ref{tab:negative-overheads}). Das bedeutet, dass die instrumentierten Funktionen schneller waren als die 
nicht-instrumentierten. Jede Messung wurde in mindestens 10 Batches durchgeführt und die Bootstrap-Konfidenzintervalle 
schließen die Null aus, die Effekte sind also statistisch signifikant.
Eine Beschleunigung durch zusätzliche Instrumentierung erscheint auf den ersten Blick paradox. Es gibt jedoch eine 
sehr plausible Erklärung für dieses Phänomen.

OpenTelemetry nutzt das \ac{OTLP}, um Telemetrie-Daten von der Lambda-Funktion an das Backend zu 
senden~\cite{OTelProtoSpec}. Das Protokoll selbst basiert zum Teil auf HTTP und wärmt bei der 
Initialisierungsphase (Init Duration) die HTTP-Stacks auf. Da bei den HTTP-Benchmarks keine 
AWS-SDKs verwendet weden, findet bei der Initialisierung kein Aufwärmen der HTTP-Stacks statt. 
Das führt zu längeren Durations bei den nicht-instrumentierten Funktionen, da die HTTP-Stacks 
erst beim ersten Request initialisiert werden müssen. Das kann auch an den Overheads der 
Init-Duration-Metrik von Node.js und Python erkannt werden, die bei den HTTP-Benchmarks absolut am 
höchsten sind (siehe Tabelle~\ref{app:boot-abs-init-cold}).

Für die Praxis sind diese negativen Overheads irrelevant, da sie nur bei den ersten Funktionsaufrufen 
auftreten (Kaltstarts).

\subsection{Kostenanalyse}
\label{subsec:kosten-analyse}
Die Kostenabschätzung (Abbildung~\ref{fig:best-prices-main}) zeigt, dass Python bei 4 von 5 Benchmarks die günstigste Sprache ist. 
Java ist nur bei der Lambdaverkettung die günstigste Option. Da die Kosten direkt proportional zur Laufzeit und Speicherzuweisung sind (GB-Sekunden), 
spiegelt dies die absolute Performance wider. Die konfigurierte Speichergröße dient als Multiplikator für den Preis pro GB-Sekunde - 
1 Sekunde Laufzeit zählt bei 1 GB Speicher als 1 GB-Sekunde, bei 2 GB Speicher als 2 GB-Sekunden. Aufgrund der langen Lebensdauer der 
Lambda-Funktionen (durchschnittlich 130 Minuten), spielt die Init Duration von 1-2 Sekunden bei Kaltstarts kaum eine Rolle für die Gesamtkosten. 
Der primäre Kostentreiber ist die Duration-Metrik der Funktion. Das kann auch anhand der Ergebnisse der Duration-Metrik erkannt werden - 
die Funktion mit der kürzesten Warmstart-Duration hat auch die niedrigsten Kosten (siehe Tabelle~\ref{app:boot-abs-duration-hot} und Abbildung~\ref{fig:best-prices-main}).

\subsection{Vergleich mit Eder et al. (2023)}

Eder et al.~\cite{10254754} untersuchten den Overhead von OpenTelemetry auf AWS Lambda ausschließlich mit Node.js. Sie fanden 
signifikante Overheads bei allen Metriken sowohl bei Kalt- als auch bei Warmstarts. Die vorliegende Arbeit bestätigt diese Ergebnisse 
nicht nur für Node.js, sondern erweitert sie um die Sprachen Java und Python. Es zeigt sich, dass der Overhead von OpenTelemetry 
bei allen 3 Sprachen, für alle Metriken signifikant ist. Er ist jedoch stark sprachabhängig. Die allgemeine Tendenz ist, dass 
Python den geringsten Overhead hat, gefolgt von Java, während Node.js den höchsten Overhead aufweist. 

\subsection{Vergleich mit Rodriguez et al. (2024)}

Rodriguez et al.~\cite{Rodriguez2024Evaluation} verglichen die absolute Performance von Java, Python und Node.js auf AWS Lambda ohne 
Tracing. Ihre Rangfolge (Python am schnellsten, gefolgt von Node.js, dann Java) wird durch die vorliegende Arbeit für alle synchronen 
Szenarien bestätigt. Die einzige Ausnahme ist die asynchrone Lambdaverkettung bei Warmstarts - da hat Java die geringste Duration, 
gefolgt von Node.js und Python. Dieses Muster ist für die Overheads unterschidlich. Generell hat Python den geringsten Overhead, 
gefolgt von Java, während Node.js den höchsten Overhead aufweist.

\section{Beantwortung der Forschungsfragen}

\subsection{Leitfrage: Sprachspezifische Unterschiede des Overheads und Implikationen}

Die Ergebnisse zeigen, dass der Performance-Overhead der OpenTelemetry-Instrumentierung zwischen den drei Sprachen 
erheblich variiert. Python weist den niedrigsten Overhead auf, Node.js hat bei Kaltstarts extrem hohe Init Duration-Overheads 
und Java liegt beim relativen Overhead im Mittelfeld. Es ist zu betonen, dass die niedrigsten Overhead-Werte nicht unbedingt die 
besten absoluten Performances bedeuten. Python hat bei den Ranglisten der Overheads nur 12 von insgesamt 25 erste Plätze, aber bei 
den absoluten Performances 24 von 25 erste Plätze. Das zeigt, dass es nicht unbedingt eine Korellation zwischen relativem Overhead 
und absoluter Performance gibt - zumindest nicht in allen Fällen und nicht proportional.
Die Implikationen für die Praxis sind, dass die Wahl der Programmiersprache einen erheblichen Einfluss auf die Performance einer 
AWS-Lambda-Funktion mit Opentelemetry-Instrumentierung hat. Python ist in allen fällen der synchronen Benchmarks die performanteste 
Option, während bei der asynchronen Lambdaverkettung Java die beste Performance und die niedrigsten Kosten bietet. 

\subsection{Teilfrage 1: Prozentualer Overhead in Java, Python und Node.js}

Die durchschnittlichen Overheads für die drei Sprachen sind:
\begin{itemize}
\item \textbf{Python:} Init Duration 288,21 \%, Duration 6,83 \%, Max Memory Used 44,47 \%
\item \textbf{Java:} Init Duration 214,48 \%, Duration 106,52 \%, Max Memory Used 59,73 \%
\item \textbf{Node.js:} Init Duration 495,03 \%, Duration 47,78 \%, Max Memory Used 74,94 \%
\end{itemize}
bei Kaltstarts und 
\begin{itemize}
\item \textbf{Python:} Duration 30,44 \%, Max Memory Used 44,70 \%
\item \textbf{Java:} Duration 21,85 \%, Max Memory Used 60,13 \%
\item \textbf{Node.js:} Duration 24,93 \%, Max Memory Used 64,34 \%
\end{itemize}
bei Warmstarts.
Die vollständige Aufschlüsselung nach Benchmarks und Metriken findet sich im Anhang (siehe Tabellen~\ref{app:raw:cold} bis~\ref{app:boot:hot}).

\subsection{Teilfrage 2: Anwendungsfälle mit den größten sprachspezifischen Unterschieden}
Die größten sprachspezifischen Unterschiede in den Overheads sind:
\begin{itemize}
    \item \textbf{Init Duration:} POST-Request – Node.js hat einen 520,70\,\% höheren Overhead als Java.
    \item \textbf{Duration:} GET-Request – Java hat einen 245,54\,\% höheren Overhead als Node.js.
    \item \textbf{Max Memory Used:} DynamoDB-Leseoperation – Node.js hat einen 43,39\,\% höheren Overhead als Python.
\end{itemize}
bei den Kaltstarts und \begin{itemize}
    \item \textbf{Duration:} GET-Request – Python hat einen 27,19\,\% höheren Overhead als Node.js.
    \item \textbf{Max Memory Used:} DynamoDB-Leseoperation – Node.js hat einen 38,26\,\% höheren Overhead als Python.
\end{itemize}
bei den Warmstarts.

Die größten sprachspezifischen Unterschiede in den absoluten Messwerten mit OpenTelemetry sind:
\begin{itemize}
\item \textbf{Init Duration:} DynamoDB-Leseoperation – Java weist einen um 3129,38,ms höheren Medianwert auf als Python.
\item \textbf{Duration:} DynamoDB-Leseoperation – Java weist einen um 1800,79,ms höheren Medianwert auf als Python.
\item \textbf{Max Memory Used:} Lambda-Verkettung – Java weist einen um 142,00,MB höheren Medianwert auf als Python.
\end{itemize}
bei den Kaltstarts und
\begin{itemize}
\item \textbf{Duration:} Lambda-Verkettung – Python weist einen um 14,73,ms höheren Medianwert auf als Java.
\item \textbf{Max Memory Used:} Lambda-Verkettung – Java weist einen um 141,00,MB höheren Medianwert auf als Python.
\end{itemize}
bei den Warmstarts.

\subsection{Teilfrage 3: Praxisempfehlungen zur Sprach- und Konfigurationswahl}
Die Ergebnisse der Arbeit legen nahe, dass die die Duration-Metrik der wichtigste Kostentreiber 
ist. Wie bereits in Abschnitt~\ref{subsec:kosten-analyse} erwähnt, haben die Funktionen mit der kürzesten Duration auch die 
niedrigsten Kosten. Daher sind die Empfehlungen für die Praxis eindeutig. Für die synchronen Benchmarks HTTP-GET, HTTP-POST, DynamoDB-Leseoperation und DynamoDB-Schreiboperation
ist Python die beste Wahl, da es die beste Performance und die niedrigsten Kosten bietet. Für die asynchrone Lambdaverkettung 
ist Java die beste Option. Außerdem hat Python mit den niedrigsten Initialisierungszeiten die schnellsten Antwortzeiten sowohl bei 
Kalt- als auch bei Warmstarts. Die Preisunterschiede zwischen den Sprachen sind für eine Million sind bei den meisten Benchmarks 
relativ gering. Das ändert sich jedoch mit wachsenden Aufrufzahlen. Eine Softwarearchitektur besteht üblicherweise aus mehr als nur 
einer Lambda-Funktion. Daher können die Kostenunterschiede bei mehreren Funktionen schnell größer werden. Um diesen Effekt zu visualisieren, 
ist auf der unten stehenden Abbildung~\ref{fig:dynamodb-cost-ch06} beispielhaft die Kostenabschätzung für die DynamoDB-Leseoperation dargestellt.

Ein wichtiger Hinweis für die Praxis ist, dass je kleiner der Traffic ist, desto weniger relevant die Wahl der Sprache für die Kosten ist. 
Anwendungen mit geringerem Traffic können daher den Kosten-Trade-off zugunsten anderer Faktoren wie Entwicklerproduktivität, Vertrautheit 
mit der Sprache, vorhandene Codebasen oder andere Aspekte eingehen.
