\chapter{Stand der Forschung}
\label{ch:related-work}

\section{Einleitung}
\label{sec:related-intro}

Die zunehmende Verbreitung serverloser Architekturen in den vergangenen Jahren hat die Notwendigkeit für die Durchführung 
von Forschungsarbeiten, die sich mit der Performance und Observability solcher Systeme beschäftigen geschaffen. 
Insbesondere die Wahl der Programmiersprache und der Einsatz von Distributed-Tracing-Tools stehen im Fokus, 
da beide Faktoren signifikanten Einfluss auf die Effizienz und Betriebskosten serverloser Anwendungen haben können. 
Während verschiedene Studien isolierte Aspekte wie Sprachperformance~\cite{Rodriguez2024Evaluation} oder Tracing-Overhead~\cite{10254754} 
untersuchen, fehlt bislang eine vergleichende Analyse, die beide Dimensionen gemeinsam betrachtet.

Dieses Kapitel stellt zwei zentrale Arbeiten vor, die für die vorliegende Untersuchung 
von besonderer Relevanz sind. Die erste Arbeit \emph{"A Comparison of Distributed Tracing Tools in Serverless Applications"}\cite{10254754} 
vergleicht den Overhead verschiedener Distributed-Tracing-Tools in AWS-Lambda-Funktionen. 
Die zweite Arbeit \emph{"Evaluation of Programming Languages for Memory Usage, Scalability, and Cold Start, 
on AWS Lambda Serverless Platform as a Case Study"}\cite{Rodriguez2024Evaluation} evaluiert die Performance 
unterschiedlicher Programmiersprachen auf derselben Plattform. 
Beide Studien werden hinsichtlich ihrer Motivation, Methodik, Ergebnisse und Limitationen analysiert, 
um darauf aufbauend die Forschungslücke für die eigene Arbeit zu addressieren. Das Zentrale Erkentnisinteresse 
stammt jedoch primär aus der ersten Studie, da die Untersuchung des Sprachfaktors im Bezug auf Tracing-Overhead  
im Abschnitt "Future Work" der Arbeit explizit als Empfehlung für weitere Forschungen erwähnt wird.

\section{Vergleichsstudie 1 – A Comparison of Distributed Tracing Tools in Serverless Applications}
\label{sec:a-comparison}

\subsection{Motivation}
\label{subsec:eder-motivation}

Die Autoren adressieren die wachsende Bedeutung von Observability in serverlosen Anwendungen. 
Da diese aufgrund ihrer feingranularen, verteilten Natur komplex und fehleranfällig sein können, 
ist Distributed Tracing ein zentrales Werkzeug zur Nachverfolgung von Anfragen. 
Gleichzeitig folgt Serverless Computing einem nutzungsbasierten Abrechnungsmodell, 
sodass der durch Tracing verursachte Overhead direkte Auswirkungen auf die Kosten haben kann. 
Die Autoren motivieren ihre Arbeit mit dem Fehlen eines detaillierten Vergleichs verschiedener Tracing-Tools 
speziell in serverlosen Umgebungen\cite{10254754}.

\subsection{Untersuchungsgegenstand und Methodik}
\label{subsec:eder-methodology}

Die Studie untersucht drei Open-Source-Distributed-Tracing-Tools - Zipkin, OpenTelemetry (mittels AWS Distro for OpenTelemetry, ADOT) 
und Apache SkyWalking. Alle Tools wurden in AWS-Lambda-Funktionen integriert, wobei Node.js 16.x als konkrete Laufzeitumgebung 
gewählt wurde. Die Evaluation erfolgte anhand von fünf Microbenchmarks, die typische Interaktionsmuster serverloser Anwendungen abbilden 
(siehe Abbildung~\ref{fig:chapter02:aws_lambda_use_cases}):

\begin{enumerate}
    \item \textbf{Function Invocation}: Eine Lambda-Funktion ruft eine weitere Lambda-Funktion auf.
    \item \textbf{HTTP GET}: Die Funktion sendet eine GET-Anfrage an einen externen HTTP-Endpunkt.
    \item \textbf{HTTP POST}: Die Funktion sendet eine POST-Anfrage an einen externen HTTP-Endpunkt.
    \item \textbf{DynamoDB Read}: Die Funktion liest einen Datensatz aus einer DynamoDB-Tabelle.
    \item \textbf{DynamoDB Write}: Die Funktion schreibt einen Datensatz in eine DynamoDB-Tabelle.
\end{enumerate}

Für jede Kombination aus Benchmark und Tracing-Tool wurden 100 warme Funktionsaufrufe 
(hot starts) durchgeführt und die Metriken Laufzeit (runtime), Speicherverbrauch (memory usage) 
und Initialisierungsdauer (initialization duration) erfasst. 
Die Tracing-Backends wurden auf einer AWS-EC2-Instanz gehostet. 
Zusätzlich wurde eine nicht-instrumentierte Baseline gemessen.

\label{subsec:overhead-results}

Zipkin verursacht in Durchschnitt 7,51\,\% zusätzlichen Speicherbedarf,

Zipkin erhöht die Initialisierungszeit um durchschnittlich 70,22\,\%, SkyWalking um 208,04\,\%

\subsection{Einordnung}
\label{subsec:eder-discussion}

Die Studie von liefert erstmals einen systematischen Vergleich von Tracing-Tools in einer serverlosen 
Umgebung. Die Ergebnisse zeigen, dass der Overhead erheblich variieren kann – Zipkin als 
framework-basiertes Tool erweist sich als leichtgewichtig, während die agent-basierten 
Ansätze von OpenTelemetry (ADOT) und SkyWalking teilweise einen signifikanten Overhead 
verursachen. Besonders auffällig ist der hohe Speicher- und Kaltstart-Overhead von ADOT, 
der von den Autoren auf den zusätzlichen Collector zurückgeführt wird.

Allerdings beschränkt sich die Untersuchung auf Node.js als Laufzeitumgebung. 
Zudem wurden nur synchrone Aufrufe betrachtet. Die Autoren selbst regen an, 
weitere Sprachen und Plattformen in zukünftige Vergleiche einzubeziehen – eine 
zentrale Motivation für die vorliegende Arbeit.

\section{Vergleichsstudie 2 – Evaluation von Programmiersprachen auf AWS Lambda (Rodriguez et al. 2023)}
\label{sec:rodriguez-2023}

\subsection{Motivation}
\label{subsec:rodriguez-motivation}

Rodriguez et al.~\cite{Rodriguez2024Evaluation} adressieren die Frage, wie
 sich verschiedene Programmiersprachen in serverlosen Umgebunge
 n hinsichtlich Initialisierung, Kaltstarts, Speichernutzung und Skalierbarkeit verhalten. Die Autoren 
 stellen fest, dass viele wissenschaftliche Arbeiten die Sprachwahyse dieser Faktoren vernachlässigen. Ziel ist es, eine fundierte Entscheidungsgrundlage für Entwickler zu schaffen, die serverlose Anwendungen auf AWS Lambda implementieren.

\subsection{Untersuchungsgegenstand und Methodik}
\label{subsec:rodriguez-methodology}

Die Studie vergleicht drei Programmiersprachen: JavaScript (Node.js), Python und Java. Als Anwendungsfall dient eine REST-API für CRUD-Operationen auf DynamoDB, die einen Fragebogen (angelehnt an den argentinischen Zensus 2022) abbildet. Die API wurde in allen drei Sprachen implementiert und mit dem Open-Source-Tool Artillery getestet. Die Experimente umfassen:

\begin{itemize}
    \item \textbf{Initialisierungstests}: Fünf Kaltstarts pro Sprache und CRUD-Operation bei 10 GB Speicherzuweisung. Zusätzlich wurden nach jedem Kaltstart sofort warme Ausführungen gemessen, um den Unterschied zwischen Kalt- und Warmstart zu quantifizieren.
    \item \textbf{Skalierbarkeitstests}: Anfragen mit steigender Parallelität (1, 10, 25, 50, 100, 200 gleichzeitige Aufrufe) zur Beobachtung des Skalierungsverhaltens der Leseoperation (Read).
    \item \textbf{Speichertests}: Variation des zugewiesenen Speichers (512 MB, 1, 2, 4, 6, 8, 10 GB) zur Analyse des Einflusses auf Laufzeit und Initialisierung (beispielhaft für die Create-Operation).
\end{itemize}

Alle Funktionen wurden mit einer reservierten Konkurrenz von 200 Instanzen betrieben, um eine hohe Parallelität zu ermöglichen und gleichzeitig kontrollierte Bedingungen zu schaffen.

\paragraph{Einfluss des Speichers}
Abbildung~\ref{fig:rodriguez-memory} (hier nicht reproduziert) zeigt die Ausführungszeiten der Create-Operation in Abhängigkeit vom konfigurierten Speicher (512 MB bis 10 GB). Reduzierter Speicher führt bei allen Sprachen zu längeren Laufzeiten, jedoch ist der Effekt bei Java besonders ausgeprägt: Unter 1024 MB steigen die Laufzeiten drastisch an, während Node.js und Python auch bei geringerem Speicher akzeptable Werte liefern. \subsection{Einordnung}
\label{subsec:rodriguez-discussion}

Die Arbeit von Rodriguez et al. bestätigt, dass Python und Node.js für typische serverlose CRUD-Anwendungen besser geeignet sind als Java, wenn Kaltstartzeiten kritisch sind. Java zeigt jedoch bei hoher Auslastung und warmen Umgebungen eine konkurrenzfähige Performance und kann bei großen Speicherkonfigurationen sogar die besten Laufzeiten erzielen. Die Studie unterstreicht die Notwendigkeit, die Sprachwahl anhand des erwarteten Lastprofils und der verfügbaren Speicherressourcen zu treffen.

Als Limitation ist anzumerken, dass nur CRUD-Operationen und nur eine AWS-Plattform betrachtet wurden. Zudem fehlt eine Analyse des Zusammenspiels mit Observability-Tools – genau diese Lücke adressiert die vorliegende Arbeit.

\section{Limitationen bestehender Ansätze}
\label{sec:limitations}

Beide vorgestellten Studien liefern wertvolle Erkenntnisse für die serverlose Entwicklung, weisen jedoch spezifische Einschränkungen auf, die die Notwendigkeit weiterer Forschung begründen.

\paragraph{Eingeschränkte Sprachvielfalt:}
Die Untersuchung von Eder et al.~\cite{10254754} beschränkt sich auf Node.js, während Rodriguez et al.~\cite{Rodriguez2024Evaluation} immerhin drei Sprachen ver
gleichen, aber ohne Berücksichtigung von Tracing-Overhead.

\paragraph{Fehlende Kombination von Sprach- und Tracing-Aspekten:}
Keine der Arbeiten untersucht den \textbf{sprachspezifischen Overhead von Tracing-Tools}. Gerade dies ist entscheidend für die Praxis, da Entwickler sowohl die Programmiersprache als auch die Observability-Strategie wählen müssen.

\paragraph{Fokus auf einzelne Metriken:}
Während Eder et al. Runtime, Speicher und Initialisierung betrachten, konzentrieren sich Rodriguez et al. primär auf Laufzeit und Skalierung. Ein umfassendes Bild, das alle relevanten Performance-Aspekte vereint, fehlt.

\paragraph{Plattformabhängigkeit:}
Beide Studien wurden ausschließlich auf AWS durchgeführt. Ergebnisse lassen sich nicht ohne weiteres auf andere Cloud-Anbieter übertragen.

\paragraph{Statistische Aussagekraft:}
Die Stichprobengrößen sind teils gering (100 Invocations pro Benchmark bei Eder et al., fünf Kaltstarts bei Rodriguez et al.), was die Verallgemeinerbarkeit einschränkt.

\section{Fazit}
\label{sec:conclusion-related}

Die betrachteten Vergleichsstudien belegen, dass sowohl die Wahl der Programmiersprache als auch der Einsatz von Distributed-Tracing-Tools erheblichen Einfluss auf die Performance serverloser Anwendungen haben. Während Python und Node.js geringere Kaltstartzeiten aufweisen, kann Java bei hoher Last und warmen Umgebungen punkten. Beim Tracing-Overhead erweist sich Zipkin als leichtgewichtig, während OpenTelemetry (ADOT) und SkyWalking teils deutliche Mehrbelastungen verursachen.

Allerdings klafft eine zentrale Forschungslücke: \textbf{Bislang existiert keine vergleichende Untersuchung, die den sprachspezifischen Overhead von Tracing-Tools in serverlosen Umgebungen systematisch analysiert}. Genau hier setzt die vorliegende Arbeit an. Sie übernimmt die validierten Microbenchmarks von Eder et al. und erweitert sie um einen Sprachvergleich (Java, Python, Node.js) sowie um eine detaillierte Messung von Runtime-, Memory- und Cold-Start-Overhead. Dadurch wird erstmals eine fundierte Entscheidungsgrundlage für die kombinierte Wahl von Programmiersprache und Observability-Strategie geschaffen.

Daraus ergeben sich drei Leitlinien für die eigene Untersuchung:

\begin{enumerate}
    \item \textbf{Sprachvergleich}: Es müssen die drei am weitesten verbreiteten Sprachen (Java, Python, Node.js) in identischen Benchmarks gegenübergestellt werden.
    \item \textbf{Reproduzierbarkeit}: Die Experimente sind so zu dokumentieren, dass sie auf anderen Plattformen wiederholt werden können.
    \item \textbf{Ganzheitliche Metriken}: Neben der reinen Laufzeit sind Speicherverbrauch und Kaltstartzeiten zu erfassen, um ein vollständiges Bild des Overheads zu erhalten.
\end{enumerate}
